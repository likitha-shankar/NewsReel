01

Executive summary

NewsReel turns a list of interests into a scheduled, two-host news podcast. A five-stage pipeline gathers news from three independent sources, enriches the top articles with real body text, drafts a script with GPT-4o, fact-checks it against the sources with a different AI provider, and synthesizes audio with ElevenLabs — all as a background job so the UI never blocks.

The brief asked for three things: a UI to set interests and customize the show, a backend that pulls news via APIs or scraping, and an internal dashboard tracking product success. All three are built, plus a working reliability layer — retries, graceful degradation, idempotency, cross-provider quality review — because a take-home is a better signal when it's built the way the real thing would need to work.

03

The generation pipeline

One episode walks five stages, each committed to the database as it completes — the stage column doubles as a lightweight state machine and a resumability anchor.

1 · News
RSS + Hacker News API + scrapers, per interest. One dead source logs and continues.
→
2 · Enrich
Top articles fetched in full, TLDR'd by gpt-4o-mini — depth-scaled (see §05).
→
3 · Script
GPT-4o writes structured JSON dialogue. Segmented per-topic above 500 words.
→
4 · QA
Gemini grades vs. the saved sources on a deduction rubric.
→
5 · TTS
ElevenLabs per line, byte-concatenated into the episode file.
Score < 7.0 → one rewrite with the reviewer's notes → best-of-two ships (never worse than the first draft). QA is advisory: a reviewer failure ships the episode unreviewed, flagged, never blocked.

Why background + polling, not a queue or sockets

Celery/Redis is the correct answer at scale, but needs a broker and worker fleet a single-user tool shouldn't carry. WebSockets are nicer UX for zero real benefit at this scale. FastAPI's own background tasks plus a four-second poll keep the UI responsive with zero extra infrastructure — and the pipeline is already one isolated function, so the Celery migration later is mechanical, not a rewrite.

04

Click to audio — the full workflow

What actually happens, in order, when a user presses Record — from the click to a playable episode.

1BrowserUser clicks Recordt = 0ms

Frontend sends POST /api/episodes with format, optional length override, focus text, and an optional article link.

2APIValidate & guardt ≈ 5ms

Pydantic checks the shape (enum format, length caps). An idempotency check looks for an episode already generating — if one exists, that episode is returned instead of starting a duplicate.

3APIInsert row, hand off, respondt ≈ 20ms

An episodes row is written with status = generating, stage = queued. The pipeline function is handed to FastAPI's BackgroundTasks. The API responds 202 Accepted with the episode id — the click is now "done" from the browser's point of view.

4BrowserStart polling

The UI adds the episode optimistically and polls GET /api/episodes/{id} every 4 seconds, rendering whichever stage comes back.

5WorkerStage: news~2–5s

Fetches per-interest results from RSS, the HN API, and scrapers, plus the listener's article link if one was given. A dead source logs and is skipped; total silence across all sources raises a clear error instead of continuing with nothing.

6WorkerStage: enrich~3–8s

The top fetchable articles per topic are read in full and TLDR'd by gpt-4o-mini — more articles and more technical detail if the listener's depth is set to expert.

7WorkerStage: script~5–10s

gpt-4o writes structured JSON dialogue. Above a 500-word target it segments — one call per topic, stitched with continuity — rather than one call that would under-deliver.

8WorkerStage: qa~2–4s

Gemini grades the script against the saved sources. Below 7.0, one rewrite runs with the reviewer's notes and the better of the two drafts ships. A reviewer failure is caught — the episode proceeds unreviewed, flagged, never blocked.

9WorkerStage: tts~20–30s

Each dialogue line is synthesized by ElevenLabs individually — with the neighboring lines passed as context for natural prosody — written to its own segment file, then byte-concatenated into the final MP3. This is where most of the wall-clock time goes.

10WorkerStatus → readyt ≈ 35–60s

The episode row is updated: audio file, duration, QA score, transcript, sources. Status flips to ready only after every line has synthesized successfully — nothing half-built is ever exposed.

11BrowserNext poll sees "ready"

The player renders and the episode is playable — and simultaneously available over the podcast RSS feed for any subscribed app.

If any stage raises — a dead-end news search, a malformed model response, an exhausted retry — the except wrapper around the whole pipeline catches it, sets status = failed, and records the error on the row. The user sees "didn't make it to air, try again"; dev mode shows the real error.

Reliability & quality gates

The part that matters most for a company running AI systems at production call volume: what happens when something breaks.

Cross-provider QA — the deepest piece

Every script is graded by Gemini 2.5 Flash while GPT-4o writes it. A model judging its own output shares its own blind spots; a different provider is an independent check. The rubric is deduction-based, not additive — the first version (additive, "8+ is airable") graded everything a 9.0. Deduction spread real scores from 2.0 to 9.0 on the same content:

IssuePenalty
Ungrounded factual claim (not traceable to a source)−3.0
Editorializing beyond what a source states−1.0
Markdown, stage directions, emojis, spoken URLs−1.0
Missing greeting or sign-off−1.0
3+ consecutive lines by one host−1.0
Tone mismatch−0.5 to −2.0
Filler with no information−0.5

Provenance backs the check: every source is written to the database before the writer model ever runs, so the sources it's graded against are raw fetched data — not something either model could have invented after the fact.

Idempotency — proven, not assumed

Both the record endpoint and the scheduler check for an in-flight episode before starting another. Verified with two rapid POST requests: both returned the same episode id, no duplicate generation, no double ElevenLabs charge.

Known, named limit: the in-flight guard is single-instance. Multi-instance would need a (user, scheduled_slot) unique constraint in Postgres — a small, specific upgrade, not a redesign.

The length bug — a debugging story worth telling in the room

Setting a 15-minute episode produced 4 minutes of audio. The instinct is "use a better model." Investigation showed the real ceiling: a single LLM completion reliably under-delivers past ~500 target words regardless of the model. The fix was segmentation — one call per topic with its own word budget, stitched with continuity between segments — which reliably hits length because no single call is asked to sustain thousands of words.

The starved-input bug — found by reading the database, not guessing

Episodes felt thin regardless of length settings. Querying the stored sources JSON showed the actual input the writer received: RSS summaries were just the title repeated, Hacker News gave engagement counts, scrapers gave nothing. No model writes a substantive script from fifteen headlines — the bottleneck was starved input, not model capability. The fix: an enrichment stage that fetches the top articles' real bodies and distills them with a cheap model before the writer ever sees them.

07

Product surfaces

A listener mode and a developer mode share one codebase; the DEV toggle flips both the visible feature set and the visual theme.

Listener-facing

  • Formats: two-host deep dive, single-narrator brief, or opposing-side debate — plus a solo-host station mode.
  • One-click show profiles: Commute (2-min brief), Daily Dive, Sunday Longread, Face-off — format, length, and steering bundled into one tap.
  • Personalization: five languages, three tones with concrete style directives, three listener-knowledge depths that now scale source depth as well as vocabulary, per-episode focus steering, and an optional listener-supplied article folded in as its own source.
  • Ask the Hosts: a question — typed or voice-dictated — answered strictly from that episode's transcript and sources, spoken back in the host's voice. An uncovered question offers to become the next episode, one click.
  • Line-level re-voicing: per-line TTS segments are kept on disk, so editing one transcript line re-synthesizes only that line — about 0.6 seconds, versus a full ~40-second regeneration.
  • Distribution: a private podcast RSS feed subscribes in any podcast app; soft delete with a 30-second undo.

Operator-facing (dev mode)

  • Dashboard: generation success rate, average latency, QA pass rate, listen rate and completion by episode length, cost per episode — the metrics that describe pipeline health, not vanity counts.
  • Console: model, temperature, and voice tuning, with model IDs verified against each provider before saving, and a hard block on setting the QA judge to the writer's own provider.
08

Bugs found & fixed

Found through deliberate audits of the shipped product, not left for a reviewer to discover.

Length
15-minute setting produced 4 minutes of audio

Root cause: single-call generation under-delivers word targets past ~500 words. Fixed with per-topic segmentation.

381 words → 841 words on a 5-minute target, measured
Content depth
Episodes read thin regardless of settings

Diagnosed by querying the database directly: the writer was receiving headlines only. Fixed with an article-enrichment stage.

10 of 15 sources upgraded from title-only to real TLDRs, verified in the DB
Crash
QA judge response truncated mid-JSON, crashing the review

The judge's output cap was hit on longer scripts. Fixed with a raised budget, an 8-issue cap on the judge's own output, and a lenient parser that salvages a closed-but-truncated array.

Data integrity
Malformed model output could ship a silent-empty episode

Nothing validated script lines before spending on audio. Added a gate that keeps only well-formed lines and fails loudly if none remain — before any TTS cost is incurred.

Security
Listener-supplied article links could target internal services

Confirmed the fetcher could read an internal API endpoint. Fixed with a guard refusing loopback, private, and link-local addresses before any request is made.

Personalization
"Expert" depth changed tone, not substance

Expert episodes sounded more technical over identical source material. Enrichment now scales with depth — more articles, longer and more technical summaries for expert listeners.

524-char TLDR (basic) vs. 935-char TLDR (expert) on the same source article

Voice synthesis dominates unit economics — a five-minute episode costs roughly $0.20–0.30 all-in; a fifteen-minute one, $0.60–1.20. That's why the length slider warns above eight minutes, and why cost-per-episode is a first-class dashboard metric rather than an afterthought.

10

What I'd do next

  1. Real event tracking behind the dashboard — an events table replacing the current seeded mock data.
  2. Concurrent TTS synthesis, batched by speaker turn rather than fully sequential — most of the latency win without losing cross-line prosody.
  3. QA calibration — sample episodes for human review and track agreement with the automated judge over time.
  4. Multi-tenancy — a user_id column, per-user scheduler jobs, and a durable (user, slot) idempotency key for multi-instance deployment.
